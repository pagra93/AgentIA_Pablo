Como Funcionan los Tokens
Los agentes NO se cargan todos a la vez (Context Engineering)
Cada agente se ejecuta en su propio contexto separado. Si /define usa 4 agentes, son 4 contextos independientes. Y /build va mas alla: spawna un sub-agente fresco (200k tokens) por cada story — la calidad es PEAK para todas, sin importar cuantas haya.
Tu solo nombras agentes cuando quieres algo especifico
Si usas /commands (/analyze, /define, /build...), los agentes se invocan SOLOS segun las instrucciones del command. No tienes que nombrarlos. Solo nombras un agente cuando quieres usarlo fuera del pipeline: "Usa el agente quality-coach para revisar estas stories."
Mas pasos = mas tokens pero mejor calidad
Para algo trivial, prompt directo. Para algo grande, pipeline completo. Tu decides el trade-off. No hay obligacion de usar todo siempre.

/saveCommit y push seguro a GitHub
No usa agentes — es un command directo con validaciones automaticas
Pipeline: Valida rama (no main!) -> Detecta secrets -> Stage -> Commit formateado -> Push
El tipo de commit se sugiere automaticamente desde el nombre de la rama: feat/* -> feat:, fix/* -> fix:, etc.
Que hace paso a paso
1. Valida rama — Si estas en main, ABORTA. Nunca commits directos a main.
2. Muestra cambios — git status + diff stats. Te pregunta si son correctos.
3. Detecta secrets — Busca .env, .pem, .key, secrets/. Advierte si encuentra algo.
4. Stage — git add .
5. Commit — Sugiere tipo (feat/fix/chore) segun rama. Pide descripcion.
6. Push — fetch + pull --rebase + push. Maneja ramas nuevas con -u.
7. Resumen — Muestra rama, commit, y proximos pasos (crear PR, merge).
Reglas de branching (rul-git-branch-management)
Todos los engineering agents (frontend-dev, backend-arch, mobile-builder, test-engineer) cargan esta regla automaticamente.

Ramas validas: feat/, fix/, chore/, refactor/, docs/, hotfix/
Formato: tipo/descripcion-corta-con-guiones (ej: feat/user-authentication)
Flujo: crear rama -> desarrollo -> push -> PR en GitHub -> squash merge -> limpiar
Merge: siempre via Pull Request, squash and merge recomendado
Despues del merge en GitHub: git checkout main && git pull && git branch -d feat/mi-rama && git fetch --prune

/design-to-prdDe disenos a stories verticales + PRDs
design-analyst (opus) lee Pencil, analiza 6 capas, genera stories verticales completas + PRDs por feature
Output: docs/working-docs/[feature]/ con stories.md (tickets verticales) + prd.md por cada feature
Cada story es un ticket completo: diseno, criterios BDD, notas tecnicas (DB/API/logica), planes de pruebas DEV/QA. Un dev coge UNA story y tiene TODO.
Las 6 capas que analiza por pantalla
1. UI — Componentes, estados (vacio/cargando/error/exito), interacciones, responsive, accesibilidad
2. DB — Tablas, relaciones, indices, volumen estimado, CRUD operations
3. API — Endpoints (method, path, auth, request/response), paginacion, filtrado
4. Logica — Validaciones, calculos, flujos condicionales, permisos por rol
5. Integraciones — Pagos, email, analytics, webhooks, servicios externos
6. Edge Cases — Red caida, datos invalidos, concurrencia, servicios caidos

El sistema tiene dos niveles: global (instalado una vez, disponible en todos los proyectos) y proyecto (especifico por proyecto). Los agentes se orquestan mediante workflows DAG.

Principios de Diseno
Analysis Informs, Never Blocks
Los agentes identifican riesgos y recomiendan. El PM siempre tiene la ultima palabra. Ningun agente puede impedir que sigas adelante.
Supervisors Observe, Never Impose
Los agentes age-sup-* (quality-coach, auditor, evaluator, optimizer) son read-only. Leen, puntuan, proponen. Nunca modifican archivos ni codigo.
Vertical Value Delivery
Cada incremento entrega valor end-to-end. Nunca se parte por capas tecnicas (frontend/backend/DB). Siempre por valor de usuario.
Evidence Over Assumptions
Toda recomendacion cita evidencia. Si no hay evidencia, se marca: "ASSUMPTION: [X]. Confidence: [high/medium/low]".
Minimum Viable Complexity
La solucion mas simple que funcione. Tres lineas similares son mejores que una abstraccion prematura.
Context Engineering (Fresh Context Per Story)
La calidad de un LLM se degrada al llenar su ventana de contexto. /build spawna un sub-agente fresco por cada story — 200k tokens dedicados. El orquestador se mantiene ligero (~15%), pasando RUTAS de archivos, no contenido. Regla anti-reflexiva: stories independientes no reciben referencias cruzadas.
Structural Verification (Must-Haves)
Cada story define Verdades (que debe ser cierto), Artefactos (que archivos deben existir con contenido real), y Conexiones (que debe estar cableado). /review verifica esto automaticamente y detecta stubs (TODO, FIXME, arrays vacios) antes de pasar a QA audit.

Los 15 Agentes — Que Hace Cada Uno y Cuando Usarlo Solo

Cada agente explicado con: que hace, cuando se activa automaticamente (en pipeline), y cuando invocarlo tu directamente.

design-analyst opus
Que hace: Lee disenos de Pencil (o screenshots/mockups), analiza 6 capas (UI, DB, API, Logica, Integraciones, Edge Cases) y las reorganiza en stories verticales completas — cada story es un ticket con: diseno (anatomia, navegacion, interaccion), criterios BDD (Given-When-Then + Scenario Outlines), notas tecnicas (DB, API, logica, integraciones, edge cases), y planes de pruebas DEV/QA. Usa formato universal kno-story-ticket-template. Aplica slicing vertical (heuristicas de Eduardo Ferro) para que cada story sea independiente, deployable, y <=3 dias.

En pipeline: Se activa con /design-to-prd.
Output: stories.md (tickets verticales) + prd.md por feature. Dos paths: Fast Track (/plan directo) o Full Pipeline (/analyze + /define).
Usalo solo cuando: "Tengo disenos y quiero stories listas para desarrollar" o "Analiza estas pantallas y genera tickets completos."
quality-guard opus
Que hace: Guardian de la puerta entre problema y solucion. Evalua PRDs en 3 dimensiones: Completitud del Problema (metricas, Gemba walk), Calidad del Proceso (AS-IS/TO-BE), y Separacion QUE/COMO (detecta contaminacion de solucion con Alternative Tool Test). Score final = MINIMO de las 3. Genera handoff/escalation documents.

En pipeline: Se activa automaticamente con /analyze (paso 1).
Usalo solo cuando: "Tengo un PRD, quiero saber si esta listo antes de empezar" — sin necesidad de correr todo /analyze.
researcher opus
Que hace: Investigador de campo con Mom Test. Detecta 3 tipos de gaps (Proceso Funcional, Inventario Secciones, Contexto Usuario). Genera guias de entrevista por ROL con preguntas que revelan verdad (nunca opinion). Produce JTBDs Reforzados con Wendel Checklist y START/STOP/DIFFERENT. Dos modos: Discover (exploratorio) y Validate (confirmatorio).

En pipeline: Se activa con /analyze (paso 2, despues del quality-guard).
Usalo solo cuando: "Necesito investigar un problema, disenar entrevistas, o hacer analisis competitivo" — incluso sin PRD formal.
jtbd-architect opus
Que hace: Traductor de research a JTBDs Reforzados. Cada JTBD tiene 8 elementos obligatorios: Job Principal, Struggle (4 capas), Trigger, Outcome, 3 Motivaciones, Anxieties/Barriers, Validacion Job-vs-Solucion (test "multiples formas"), y Rastreo de Fuente. Aplica Wendel Checklist y mapea Behavior Change con 3 rangos (Minimo/Target/Over-top).

Modo Enrich: Si recibe stories de /design-to-prd, mapea cada JTBD a una story existente (mapping table JTBD → Story ID) para que el story-writer sepa QUE enriquecer con QUE.

En pipeline: Se activa con /define (paso 1).
Usalo solo cuando: "Tengo research/entrevistas y quiero generar JTBDs estructurados" — sin correr todo /define.
story-writer opus
Que hace: Convierte JTBDs en stories deployables. Estructura: Epic > Feature > Story. Cada story tiene: narrativa con job performer especifico, Behavior Change (START/STOP/DIFFERENT cuantificado), Given-When-Then testable, Scoring 6D, y metricas de exito en 3 rangos. Retiene TODA la informacion del research sin colapsar.

Modo Enrich: Si hay stories de /design-to-prd, las enriquece en vez de crear nuevas — upgrade Historia+Definicion de [DERIVADO] a JTBD real, preserva Diseno+Notas tecnicas, merge Criterios, recalcula Scoring 6D.

En pipeline: Se activa con /define (paso 2, despues de jtbd-architect).
Usalo solo cuando: "Tengo JTBDs y quiero convertirlos en stories listas para sprint."
story-builder opus NUEVO
Que hace: Agente AUTONOMO que toma una idea/problema del PM y produce un story ticket completo en formato universal (kno-story-ticket-template). Trabaja 7 fases internas sin interrogar al PM: (1) detecta trampa de solucion, (2) deriva Job con Tecnica del Por Que, (3) Wendel Checklist, (4) 3 dimensiones del job, (5) Behavior Change con START/STOP/DIFFERENT, (6) compila ticket completo (diseno [DERIVADO], criterios BDD, notas tecnicas [DERIVADO], planes pruebas, scoring 6D), (7) recomienda crear diseno en Pencil (opcional). Explica POR QUE tomo cada decision (efecto formativo). Gaps se marcan como [GAP] o [HIPOTESIS] — nunca bloquea.

En pipeline: Se activa con /story.
Usalo solo cuando: "Tengo una idea y quiero un ticket completo" o "Convierte esto en una user story."

Flujo iterativo: /story → story ticket → disena en Pencil → /design-to-prd → stories con diseno real → /plan → /build (contexto fresco/story)
quality-coach opus SUPERVISOR
Que hace: Entrenador de Calidad. Evalua stories como EXPERIMENTOS sobre cambio de comportamiento, no como checklist. Diagnostica en 6 dimensiones con 2 hard rules (sin Wendel = max 5, sin cuantificar = max 5). Detecta 7 antipatrones. Para cada story debil, propone reescritura en JTBD. Detecta patrones entre stories ("3 de 5 tienen Usuario Fantasma"). NUNCA modifica — solo diagnostica y sugiere.

En pipeline: Se activa con /define (paso 3, despues del story-writer).
Usalo solo cuando: "Tengo stories (mias o de otro) y quiero saber si son buenas" — evalua cualquier story, no solo las generadas por el sistema.
story-splitter sonnet
Que hace: Descompone stories grandes (>3 dias) en incrementos verticales usando 9 heuristicas de Eduardo Ferro. Cada split: independientemente valioso, deployable, ≤3 dias. SIEMPRE vertical (end-to-end), NUNCA horizontal (por layer). El primer split es el "experimento sobrevivible."

En pipeline: Se activa con /define (paso 4, solo si hay stories >3 dias).
Usalo solo cuando: "Esta story es demasiado grande, necesito partirla."
tech-architect opus
Que hace: Disena la arquitectura mas simple que cumpla las stories. Lee CLAUDE.md primero (stack non-negotiable). Genera ADRs (Architecture Decision Records) con alternativas consideradas. Identifica componentes, data flow, y riesgos. Tambien ejecuta un scan de unknown-unknowns sobre la arquitectura.

En pipeline: Se activa con /plan (paso 1).
Usalo solo cuando: "Necesito disenar la arquitectura de esta feature" o "Necesito un ADR para esta decision."
sprint-planner sonnet
Que hace: Prioriza stories por Value/Effort, verifica Definition of Ready, crea sprint plan con dependencias y oportunidades de paralelizacion. Buffer 20%. Primera story = mayor riesgo (fail fast). Escribe el plan a tasks/todo.md.

En pipeline: Se activa con /plan (paso 2, despues del tech-architect).
Usalo solo cuando: "Tengo stories listas y necesito priorizarlas en un sprint."
Engineering — Context Engineering v3.0
No hay agentes de engineering especializados. Claude Code ya sabe programar — lee CLAUDE.md para el stack. Los agentes de frontend/backend/mobile eran redundantes y se eliminaron.

/build usa Context Engineering: el orquestador lee SOLO el sprint plan (metadata) y spawna un sub-agente fresco por cada story. Cada sub-agente tiene su propia ventana de 200k tokens → calidad PEAK para cada story, sin importar cuantas haya en el sprint.

Antes: Claude implementaba todo en una sesion. Despues de 3-4 stories, el contexto se llenaba y la calidad caia.
Ahora: Cada story tiene contexto fresco. El orquestador se mantiene ligero (~15%). Si se interrumpe, retoma con build-state.md.

Regla anti-reflexiva: Stories independientes no reciben referencias de otras stories. Solo se pasan dependencias reales (tipos compartidos, APIs).

Como Acumulan Conocimiento

qa/qa-report.md es append-only. Cada /review anade una seccion nueva. El Optimizer lee TODAS las entradas anteriores para detectar tendencias:

"Error handling aparecio en 3 de las ultimas 5 auditorias"
"Quality score tendencia: 6.2 -> 7.1 -> 7.8"
"Propuesta: anadir rul-api-error-handling"
El Optimizer PROPONE. El PM DECIDE. Nunca se aplican cambios automaticamente.

Los workflows definen el orden exacto de ejecucion usando formato DAG (Directed Acyclic Graph). Cada paso puede depender de otros, recibir datos de pasos anteriores, y ejecutarse condicionalmente.

Formato DAG
depends_on: [step_id] — este paso espera a que otro termine
${{ steps.X.outputs.Y }} — recibe datos del output de otro paso
conditions: — solo se ejecuta si la condicion es true
agent: / skill: — que agente o skill ejecuta este paso

Los commands/ (*.md) son la interfaz: instrucciones en lenguaje natural que Claude Code entiende. Los workflows/ (*.yaml) son la especificacion formal: DAG con dependencias y data flow. Ambos describen lo mismo de distinta forma.

Tu EscribesCommand (lenguaje natural)Workflow (DAG formal)

/definecommands/define.mdworkflows/wor-define.yaml
/buildcommands/build.mdworkflows/wor-build.yaml
/reviewcommands/review.mdworkflows/wor-review.yaml
/savecommands/save.mdworkflows/wor-save.yaml

Cuando NO usar agentes
Tareas de menos de 30 segundos: prompt directo a Claude Code
Cambios triviales (typo, rename, one-liner): no necesitas /analyze -> /define -> /plan
Preguntas rapidas: "que hace este archivo?" — no necesita agente
Solo usa agentes para trabajo sustancial

Tu rutina diaria con el sistema.

Al empezar el dia
Abre Claude Code en tu proyecto
Claude lee CLAUDE.md automaticamente
Revisa tasks/todo.md para ver donde quedaste
memory/MEMORY.md y tasks/lessons.md se consultan al inicio
Al trabajar en una feature
/analyze — evalua el problema primero
/define — genera stories solo si el analisis salio bien
/plan — antes de picar codigo, ten un plan en tasks/todo.md
/build — sub-agente fresco por story (context engineering)
/save — commit y push tras cada story completada
/review — verifica ANTES de merge
Al arreglar un bug
/hotfix — pipeline corto + SIEMPRE guarda el aprendizaje (causa raiz, solucion, prevencion)
No necesitas /analyze ni /define para bugs
/hotfix ahora guarda en tasks/lessons.md automaticamente
Cuando resuelves algo dificil
/learned — guarda el aprendizaje EN EL MOMENTO, no esperes a /review
Bug de 2 horas, edge case raro, descubrimiento sobre una API, error a evitar
Se guarda en tasks/lessons.md + memory/MEMORY.md — los agentes lo veran
La proxima vez que pase algo similar, lo tendras documentado
Para tareas rapidas (<30 seg)
NO uses agentes. Prompt directo a Claude Code
El sistema es para trabajo sustancial, no para one-liners
Para no perder contexto (Context Engineering)
/build usa sub-agentes frescos por story — la calidad es PEAK para todas
Si /build se interrumpe, retoma automaticamente con build-state.md
/compact al 50% de contexto en sesiones manuales — evita perder calidad
Commit despues de cada tarea completada
docs/PROJECT_KNOWLEDGE.md se actualiza con cada /review
memory/MEMORY.md acumula patrones del Code Reviewer y Optimizer
Para guardar trabajo (Git)
/save — valida rama, detecta secrets, commit formateado, push
Nunca hagas commit directo a main. Siempre ramas: feat/, fix/, chore/
Despues de /save: crea PR en GitHub, squash and merge, limpia rama local
Si estas en main por error: git checkout -b feat/nombre antes de /save
Antes de merge
/code-review si quieres solo revision
/review si quieres el pipeline completo de QA
El auditor verifica Definition of Done automaticamente
Cuando vuelves tras dias o semanas
Claude lee CLAUDE.md -> docs/PROJECT_KNOWLEDGE.md
